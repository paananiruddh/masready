### Article starts below

If you ask most IBM Maximo administrators where their time goes, it is rarely on the fun part.

It goes into untangling overlapping security groups, onboarding users into profiles that no longer make sense, and explaining why licensing feels more expensive than it should.

That becomes even harder when the conversation shifts to MAS 9.

## Why this matters

In many Maximo environments, security groups have grown over time through projects, urgent fixes, and temporary exceptions that never really disappeared.

The result is familiar:

* Too many groups
* Too much overlap
* Harder onboarding
* Harder audits
* Harder licensing conversations

This is exactly the problem MASReady is targeting next.

## What MASReady is adding

MASReady is adding a read-only capability called **Security Profiles**.

The goal is simple: help teams understand how their Maximo access model is shaped today, without changing anything in production.

It focuses on three things:

1. Baseline roles
2. Group sprawl and overlap
3. What-if scenario planning

## Baseline roles and group sprawl

Most environments have plenty of security groups, but very little role clarity.

Security Profiles is designed to surface friendlier baseline roles such as:

* Planner / Scheduler
* Supervisor
* Mobile Technician
* Report-only User
* Integration Operator
* Admin / Configuration

These are not new groups. They are a clearer way to describe the access patterns already present in the environment.

At the same time, the feature helps teams see where complexity lives:

* Groups that look very similar
* Groups with broad access but few users
* Environments that have drifted into unnecessary complexity

## What-if, without the risk

Most teams already know the kinds of questions they want to ask:

* What if these groups were combined?
* What if this cohort moved to a leaner role?
* What if admin-style access was split out into a smaller overlay?

Security Profiles is being built as a read-only planning surface for exactly that reason.

The output is a review pack, not a change script. It is meant to support governance, design workshops, and licensing conversations without touching production security.

## What this is not

This is not an automation engine.

It will not rewrite Maximo security, make hidden API changes, or ask customers to trust a black box.

It is intended to be an evidence-backed, read-only way to make security design easier to understand before complexity carries forward into MAS 9.

## What comes next

The first iteration inside MASReady will focus on four surfaces:

* Security Profiles Overview
* Baseline Roles
* Group Sprawl
* Read-only What-if Scenarios

The aim is straightforward: help teams move from group sprawl to role clarity before they take the next step into MAS 9.
