Stop making this confusing.

The MASReady public website must be **fully pre-rendered**.

Do not rely on “React SPA handles all routes” as the final answer.

The live app can still hydrate as React after load, but every public route must have static pre-rendered HTML generated at build time.

## Core rule

All public MASReady website routes must be pre-rendered.

React interactivity is fine after hydration.

But the initial HTML for every public page must exist at build output time.

## Source of truth

The source of truth remains:

```text
artifacts/masready/src/
```

Build output remains:

```text
artifacts/masready/dist/public/
```

Do not use stale root `.html` files as the source of truth.

Files like these are stale/legacy/SEO-only and must not drive the current site:

```text

demo-walkthrough.html
launch.html
architecture.html
contact.html
docs.html

```

If they remain, document them as legacy only.

## Required architecture

Implement a proper pre-render build step for the React/Vite/wouter app.

At build time, generate static HTML files for every public route.

Expected output structure should be like:

```text
artifacts/masready/dist/public/index.html
artifacts/masready/dist/public/mas9-power/index.html
artifacts/masready/dist/public/industry-previews/index.html
artifacts/masready/dist/public/industry/utilities/index.html
artifacts/masready/dist/public/demos/industries/utilities/index.html
artifacts/masready/dist/public/demo-walkthrough/index.html
artifacts/masready/dist/public/contact/index.html
```

or an equivalent static-hosting-compatible folder structure.

The important point: every route has its own generated HTML.

## Query routes

The following URLs must work correctly:

```text
/launch?mode=persisted
/preview-studio?industry=utilities
```

Static hosting may not create separate query-string HTML files. That is fine.

But the base routes must be pre-rendered:

```text
/launch
/preview-studio
```

and after hydration they must correctly read the query string and display the correct state.

Verify both query URLs directly.

## Required implementation options

Use the least disruptive reliable method for Vite + React + wouter.

Acceptable approaches:

1. Custom Node prerender script using React server rendering.
2. Vite-compatible static pre-render plugin.
3. Deterministic crawler-based prerender after build, only if it generates stable route HTML.
4. Another clean static generation approach that works with the existing app.

Do not switch the whole project to another framework unless absolutely necessary.

## Public demo safety must remain

Pre-rendering must not break demo safety.

Confirm public demo mode still enforces:

```text
PUBLIC_DEMO_MODE=true
```

Public routes must still have:

* no live Maximo API execution
* no Maximo connection test
* no API key entry/use
* no editable Maximo URLs
* no write/update/delete actions
* seed data only
* integrations visible but disabled
* no secrets exposed

No hydration warnings.

No duplicate content.
