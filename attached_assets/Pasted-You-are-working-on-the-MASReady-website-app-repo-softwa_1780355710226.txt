============================================================
OBJECTIVE
============================================================

The public synthetic preview must not be an always-open permanent URL.

Current concern:
If users can directly access URLs like:

/preview/rail
/preview/water
/preview/utilities

then they can keep using the synthetic demo forever without MASReady permission or any auditability.

Required new model:
Users must first create a short-lived synthetic preview session.

The system creates a UUID/token-based preview URL.

Example:

/preview-session/9f3a7c21-5d84-4a9c-9f88-8b62a4b1d1ef

or:

/preview/rail?session=9f3a7c21-5d84-4a9c-9f88-8b62a4b1d1ef

The session must expire after 2 hours.

After 2 hours:
- user cannot access the preview anymore
- show expired session screen
- offer buttons:
  - Start New Preview
  - Request Persisted Demo

Also write audit events for:
- session created
- industry selected
- preview opened
- page/section opened where practical
- persisted demo CTA clicked
- session expired
- preview reset/cleared

This must feel professional, secure, SEO-friendly, and enterprise-grade.

============================================================
IMPORTANT ARCHITECTURE NOTE
============================================================

The repo appears to be a GitHub Pages static site / SPA.

A purely static GitHub Pages app cannot securely enforce server-side 2-hour expiry because users can manipulate browser state.

Therefore implement this in two phases:

PHASE 1 — STATIC-SAFE MVP
Implement client-side TTL using signed-looking UUID/session metadata in browser storage.
This improves UX and audit simulation but is not security-grade.

PHASE 2 — SERVER-ENFORCED READY DESIGN
Add a clean abstraction/interface so later Azure Function / API / Worker can enforce expiry server-side.

The implementation must clearly separate:
- client preview state
- audit event writer
- session validation adapter

This allows later replacement of local/session storage with real backend validation.

Do not pretend client-side expiry is real security.
Use wording:
“Public previews are time-limited and session-scoped. Enterprise/private demos use controlled access.”

Do not say:
“cryptographically secure access control”
unless a backend is actually implemented.
