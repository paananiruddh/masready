The suite must be manifest-driven.

1. public-pages.json

This should list all public website routes/pages to test.

Example:

[
  { "path": "/", "name": "Home", "required": true },
  { "path": "/mas9-power.html", "name": "MAS9 Power", "required": true },
  { "path": "/launch.html", "name": "Launch", "required": true },
  { "path": "/architecture.html", "name": "Architecture", "required": true },
  { "path": "/demo-walkthrough.html", "name": "Demo Walkthrough", "required": true },
  { "path": "/madd/", "name": "MADD", "required": false },
  { "path": "/overpayingforai-arena/", "name": "OverpayingForAI Arena", "required": false }
]

The test must:
- Iterate every entry.
- Visit each page.
- Assert status is not 404.
- Assert title or body exists.
- Assert no console errors.
- Assert no obvious broken layout markers.
- Screenshot only on failure.

If a page is removed intentionally, developer updates public-pages.json. The test suite shrinks automatically.

If a page is added, developer adds it to public-pages.json. The suite grows automatically.

3. feature-manifest.json

Create a feature registry.

  {
    "id": "madd-arena",

    "enabledInDemo": true,
    "routes": ["/overpayingforai-arena/"],
    "expectedPublicCopy": ["Rapid Rounds", "OpenAI reviewer", "Claude reviewer"]

feature-manifest.spec.ts must:
- Loop through features.
- Visit relevant routes.
- Validate expected public copy exists if feature is enabled.
- Validate forbidden public copy does not appear.
- Allow disabled features to assert disabled messaging.

As features are added/removed, feature-manifest.json changes and the suite grows/shrinks automatically.
