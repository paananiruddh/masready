# Generic Arena Branding Guidance

## Decision

Do not tie badges, supporter labels, leaderboards, or public identity too tightly to MADD.

MADD is the first Arena use case, but tomorrow the same Arena may run:

- coding competitions
- writing competitions
- research comparisons
- design challenges
- enterprise architecture battles
- tool evaluations
- other domain assistants

## Brand hierarchy

Use this hierarchy:

```text
OverpayingForAI Arena = parent brand
MADD = first real-world use case / Season 1 / Maximo track
MASReady = Maximo delivery/product context
```

## Badge naming

Prefer generic badges:

- Arena Founding Supporter
- OverpayingForAI Founding Supporter
- Foundry Supporter
- Rapid Rounds Supporter
- Skill Pack Sponsor
- Tool Bench Sponsor
- Human-in-the-Loop Patron

Avoid over-specific badges:

- MADD-only Founder
- MADD-only Patron
- Maximo-only Supporter

## Supporter wall structure

Use generic categories:

```text
Founding Supporters
Arena Sponsors
Skill Pack Sponsors
Track Sponsors
Anonymous Supporters
```

Then optionally tag the supported track:

```text
Track: MADD / Maximo Diagnostics
Track: Coding Arena
Track: Writing Arena
Track: Research Arena
```

## CTA wording

Use:

```text
Support the Arena
Fund the next Rapid Round
Become an Arena Founding Supporter
Help build the next proof point
```

Use MADD-specific wording only when the page is clearly inside the MADD/Maximo context.

## Page implications

- `/madd-foundry/` can talk about MADD, but badges should say Arena/Foundry supporter.
- `/madd-supporters/` should be renamed or reframed later as `/arena-supporters/`.
- `/overpayingforai-arena/` is the parent Arena page.
- Future pages should use `/arena/<track>/` style routing if possible.

## Core principle

Do not sell a badge for one bot.
Sell participation in a repeatable public AI evaluation and build system.
