**Introducing the Maximo Delivery Automation Workbench.**

A locally running, Maximo-specific skilled bot architecture built around curated offline skill packs, customer data-mode boundaries, and evidence-based delivery workflows.

This is not a generic chatbot wrapper. It is not uncontrolled automation. It is a structured delivery workbench designed for Maximo teams who need to move faster without burning things down.

---

**What it actually does:**

→ Reads customer context — Maximo version, installed customisations, patch profile, integration footprint
→ Matches curated offline skill packs against delivery requirements — not generic AI, Maximo-specific patterns
→ Analyses patch and iFix impact across your real customisation surface
→ Plans AppPoint and named user licence usage — for planning visibility only, not compliance advice
→ Generates review-ready outputs — reports, checklists, runbooks, executive summaries
→ Supports multiple delivery scenarios — MAS 9 Energy & Utilities, Maximo 7.6.x Upgrade Readiness, MAS 9 Contract Mobilisation

**The safety model:**

→ No SQL execution
→ No Maximo mutation
→ No Jira mutation
→ No Azure DevOps mutation
→ Review-only output — your team decides what gets actioned
→ RBAC and customer access controls remain the real security boundary
→ Backend-only secrets — no keys in the frontend, ever

---

**Customer data boundary models:**

The public simulator uses fictional demo fixtures only. Real customers choose a data model:

→ No-store / zero-retention runtime — nothing persisted
→ Customer-hosted — data stays inside the customer's environment
→ BYO storage — customer-owned S3, Azure Blob, or GCS
→ Azure-hosted customer subdomain — governed per-customer demo or live environment
→ Controlled managed demo workspace — sales demos and prospect walkthroughs, not production data

Demo fixtures are demo-only. Always.
